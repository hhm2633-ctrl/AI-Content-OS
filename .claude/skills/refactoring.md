# Refactoring Skill

## Purpose

AI-Content-OS에서 리팩토링(코드 정리, 구조 개선, 중복 제거)을 수행할 때
프로젝트의 구조적 안정성을 지키기 위한 규칙을 정의한다.

## 규칙

- **기존 구조 최대 유지**: 현재 동작하는 WorkflowEngine 파이프라인 순서와 모듈 조합을 유지한다. 리팩토링은 "더 낫게 다시 설계"가 아니라 "기존 구조 안에서 정리"를 의미한다.
- **WorkflowEngine 변경 금지**: `src/workflow_engine.py`의 모듈 호출 순서, `WorkflowEngine.run()`의 흐름은 명시적으로 지시받지 않는 한 손대지 않는다.
- **폴더명 변경 금지**: `modules/<name>/`, `storage/<name>/` 등 기존 폴더 이름을 임의로 바꾸지 않는다.
- **모듈명 변경 금지**: `TrendCollectorModule`, `ContentModule`, `CardNewsModule` 등 기존 모듈/클래스 이름을 임의로 바꾸지 않는다.
- **클래스명 변경 금지**: 위와 동일하게, 이미 다른 모듈에서 import하고 있는 클래스명은 변경하지 않는다.
- **함수명 변경 금지**: 다른 모듈이 호출하는 `def` 이름(특히 `run()`처럼 `WorkflowEngine`이 직접 호출하는 진입점)은 바꾸지 않는다. 내부에서만 쓰이는 private 헬퍼(`_xxx`)를 새로 쪼갤 때도 기존 public 메서드 이름은 그대로 유지한다.
- **파일 이동 최소화**: 이미 존재하는 파일을 다른 폴더로 옮기지 않는다. 새 기능은 기존 파일 옆에 새 파일을 추가하는 방식으로 넣는다 (`large_implementation.md`의 "새 모듈 생성" 참고). 파일 이동이 꼭 필요하면 그 자체를 별도로 명시적 승인받는다.
- **함수 일부 수정 안내 금지**: "이 함수의 이 줄만 바꾸면 됩니다" 같은 부분 수정 안내를 하지 않는다. 리팩토링 대상 함수/파일은 항상 전체를 다시 작성해 제공한다 (`large_implementation.md` 참고).

## 리팩토링 전 확인 사항

1. 이 코드를 다른 모듈이 import하고 있는가? (`Grep`으로 사용처 확인)
2. `WorkflowEngine`이 이 모듈을 어떤 시그니처로 호출하는가?
3. 기존 fallback 동작(예외 시 안전한 기본값 반환)이 그대로 유지되는가?

## 리팩토링 후 확인 사항

- `py -m compileall src modules scripts`로 문법 오류가 없는지 확인한다.
- 변경된 모듈을 사용하는 다른 모듈의 호출부가 여전히 유효한지 확인한다.
- 가능하면 `review.md` 체크리스트를 따른다.

## 예외

사용자가 명시적으로 "모듈명을 바꿔라", "폴더 구조를 다시 짜라"고 지시한 경우에만
이 규칙을 벗어난 작업을 수행하며, 이 경우에도 변경 범위를 최대한 좁게 유지한다.
